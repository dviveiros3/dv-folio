# Trinity Proof Capture Log

## Purpose
This log separates real proof from ideas, samples, and hypotheses. It prevents Trinity from overclaiming while giving us a place to capture useful signal from distribution activities.

## Operational Resources
Use these while capturing signal to ensure alignment with the distribution strategy:
- [Warm Outbound Pack](./week-1-warm-outbound-pack.md): Use for transitioning from POV to qualified conversations.
- [Objection Bank](./growth-audit-objection-bank.md): Record exact buyer language and refine offer responses.

## Proof Categories

| Category | Meaning | Public Use |
| --- | --- | --- |
| Verified | Confirmed with source evidence | Can be used directly |
| Observed | Seen in current work but not yet externally validated | Use carefully with context |
| Anecdotal | Reported by a person or from a conversation | Use as qualitative signal |
| Simulated | Fictional or sample data | Must be labeled simulated |
| Hypothesis | Intended outcome or belief to test | Frame as hypothesis |

## Weekly Capture Template
*Copy and paste this row into the log below after each distribution or outbound session.*

**Template:**
`| [YYYY-MM-DD] | [Fill: Published Artifact Proof] | [Fill: Qualified Conversation Signal] | "[Fill: exact buyer phrasing]" | [Fill: Growth Audit Offer Change] | [Fill: Proof Confidence Level] | [Fill: Source Link / Ref] |`

## Capture Fields

Use these fields before changing public copy. The goal is to preserve the evidence behind every Growth Audit claim.

### Published Artifact Proof
- Record the artifact shipped: post, lab note, sample deliverable, diagnostic, visual, or workflow.
- Link to the source file, public post, screenshot, or commit.
- Label whether the artifact is inspectable by a buyer today.

### Qualified Conversation Signal
- Record the buyer profile: founder, SME, operator, investor, or implementation buyer.
- Note which Distribution OS loop they reacted to: POV, Proof, Conversations, Offers, or Conversion.
- Separate vanity engagement from a real buying signal.

### Exact Buyer Objection Language
- Preserve exact wording when possible.
- Use the [Objection Bank](./growth-audit-objection-bank.md) to map the objection to response language.
- Do not translate the objection into internal jargon until the exact phrase is captured.

### Growth Audit Offer Change
- Record the specific change triggered by the signal: headline, deliverable, CTA, intake requirement, proof language, implementation option, or pricing hypothesis.
- If no offer change is justified, write `No change`.
- Never upgrade a claim from hypothesis to observed without a source.

### Proof Confidence Level
- Choose one category from the table above.
- Keep sample deliverables labeled `Simulated`.
- Keep conversation-derived evidence labeled `Anecdotal` unless there is source evidence.

### Source Link / Screenshot Reference
- Link to a URL, local artifact, issue, PR, screenshot, or notes file.
- If the source cannot be shared publicly, write `Private source - do not quote publicly`.
